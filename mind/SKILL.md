# Mind — second-brain catalog

Home: `/Users/arthurgrau/Library/CloudStorage/Dropbox/Apps/mind/`
- `index.md` — the catalog. One pipe-separated line per item (format documented in its header).
- `locations.md` — registered library roots (`key | name | absolute path`). Never scan anything not listed here.
- `inbox/` — flat capture folder; Arthur dumps files here, no subfolders.
- `mind.html` + `mind-data.js` — local search frontend. The HTML is static; only `mind-data.js` is regenerated.

**Context discipline (the point of this design):** never bulk-read library
contents. Search means grepping `index.md`. Only open an actual file when
(a) cataloging it for the first time, or (b) the user asks about a specific
matched item.

## /mind find <query>

1. Grep `index.md` case-insensitively. Try the user's words first, then
   obvious synonyms/stems (e.g. "zoning" → also "land use", "planning").
   A line matches if it contains any strong term; rank lines matching more
   terms higher.
2. Report matches as: **Title** — description, tags, and the clickable
   absolute path (join location root from `locations.md` + relative path).
3. If nothing convincing matches, say so, then offer exactly two follow-ups:
   deep-grep the text-like files (`.md .txt .csv`) inside registered
   locations for the terms, or catalog first if `/mind catalog` hasn't run
   recently. Don't deep-grep without asking — it's the expensive path.
4. Never invent items. If the index is empty, say to run `/mind catalog`.

## /mind catalog

1. Read `locations.md`. For each location, list files (skip hidden files,
   skip `mind.html`/`mind-data.js`/`index.md` themselves).
2. Diff against paths already in `index.md` → the new-file list. Also note
   indexed paths whose files no longer exist; remove those lines.
3. If more than ~40 new files, tell Arthur the count and catalog in batches
   of ~40, reporting progress between batches. Continue until done unless
   he stops you.
4. For each new file, produce one index line:
   - `.md/.txt/.rtf/.csv` — read it (first ~100 lines is enough for big files).
   - `.pdf` — Read pages 1–3 only.
   - images (`.png .jpg .jpeg .svg`) — Read the image (OCR/describe).
   - media/other (`.mp3 .mp4 .mov .key .numbers .docx`) — catalog from
     filename, extension, and folder context only; note "(not opened)" in
     the description.
   - Title: the document's own title if evident, else a cleaned-up filename.
   - Description: one line, ≤140 chars, in plain language — what it is and
     what someone would remember about it.
   - Tags: 2–5 lowercase topical tags. Reuse existing tags from the index
     when they fit; don't mint near-duplicates (#urbanism vs #urban-planning).
5. Append lines to `index.md` below the `<!-- items below this line -->`
   marker. Never reorder or rewrite existing lines except for deletions
   from step 2.
6. Regenerate `mind-data.js` (see below), then report: N added, N removed,
   total items.

## Regenerating mind-data.js

Parse every item line of `index.md`, resolve absolute paths via
`locations.md`, and overwrite `mind-data.js` with exactly:

```js
// Generated by /mind catalog — do not edit by hand.
const MIND_GENERATED = "<YYYY-MM-DD HH:MM>";
const MIND_ITEMS = [
  {loc:"<key>",locName:"<display name>",path:"<absolute path>",date:"<date-added>",title:"<title>",desc:"<description>",tags:["tag","tag"]},
  ...
];
```

Escape quotes/backslashes in strings. Tags in the array drop the `#`.
Write it with a script (python3 or node) rather than by hand when there
are more than a handful of items.

## /mind add <path>

Verify the folder exists, pick a short unused key, append a line to
`locations.md`, and suggest running `/mind catalog`.
